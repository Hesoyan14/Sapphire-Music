# Sapphire в интернете: не только «на своём ПК»

## Важно понять заранее

**Яндекс.Музыка** — это огромный **лицензированный каталог** (договоры с правообладателями), CDN, приложения под все платформы и своя инфраструктура.

**Этот проект (Sapphire)** — **ваш собственный веб-плеер**: пользователи заходят на **ваш сайт**, **входят под учёткой**, слушают **файлы, которые вы или они загрузили** (MP3/WAV). Это ближе к **своему облачному плееру** или **сервису для друзей/команды**, а не к магазину подписок.

Если вам нужен именно каталог «как у Яндекса» с миллионами треков — это **другой продукт и другая правовая модель**; Sapphire для этого не подходит.

Ниже — как сделать так, чтобы Sapphire **работал как сайт в интернете** (домен, HTTPS, сервер 24/7).

---

## Что вам понадобится

| Компонент | Зачем |
|-----------|--------|
| **VPS или хостинг с Linux** | Сервер, который круглосуточно крутит Python-приложение. Обычно это Ubuntu на Timeweb, Selectel, Hetzner, DigitalOcean и т.п. |
| **Доменное имя** | Чтобы открывать `https://music.ваш-сайт.ru`, а не набор цифр IP. |
| **HTTPS** | Обязательно для логина и cookies. Делают **Caddy** или **nginx** + Let's Encrypt (бесплатные сертификаты). |
| **PostgreSQL** | Для нескольких пользователей и стабильности лучше, чем один файл SQLite на диске. Задаётся **`DATABASE_URL`**. |
| **Объектное хранилище (по желанию, но желательно)** | S3, Cloudflare R2, MinIO: переменные **`S3_*`**. Иначе файлы лежат в папке **`uploads/`** на диске VPS — для маленького проекта можно, для роста удобнее S3. |

На **Windows-сервере** в продакшене чаще используют **Waitress** вместо gunicorn; на **Linux** для Sapphire удобно **gunicorn** (см. ниже).

## Переменные окружения (минимум для «боевого» сайта)

| Переменная | Обязательно | Смысл |
|------------|-------------|--------|
| **`FLASK_SECRET_KEY`** | Да | Длинная случайная строка. От неё зависят сессии (вход в аккаунт). Без смены дефолта любой может подделать сессию. |
| **`DATABASE_URL`** | Настоятельно рекомендуется | URL PostgreSQL, например `postgresql://user:pass@localhost:5432/sapphire` |
| **`SESSION_COOKIE_SECURE`** | Да при HTTPS | Значение `true` — cookie только по HTTPS. |
| **`SAPPHIRE_BEHIND_PROXY`** | Да за nginx/Caddy | Значение `true` — учитывать заголовки `X-Forwarded-*` (правильные редиректы и схема https). |
| **`S3_BUCKET`**, **`S3_ACCESS_KEY`**, **`S3_SECRET_KEY`** | По желанию | Файлы в облаке; опционально **`S3_ENDPOINT`** для R2/MinIO. |
| **`SAPPHIRE_ALLOW_REGISTRATION`** | Если нужны саморегистрации | `true` — открыть **`POST /api/auth/register`** (иначе только вы создаёте пользователей или первый `admin`). |
| **`SAPPHIRE_BOOTSTRAP_PASSWORD`** | При первом запуске | Пароль первого пользователя **`admin`**, если база пустая. |

Секрет для ключа можно сгенерировать, например:

```bash
python -c "import secrets; print(secrets.token_hex(32))"
```

## Запуск приложения на сервере (Linux)

1. Установите Python 3, PostgreSQL (или используйте облачную БД).
2. Склонируйте проект, создайте виртуальное окружение, установите зависимости:

```bash
python3 -m venv .venv
source .venv/bin/activate
pip install -r requirements-prod.txt
```

3. Задайте переменные окружения (через `.env` + systemd, или панель хостинга — как принято у провайдера).
4. Примените миграции, если у вас они есть; если нет — при первом запуске приложение создаст таблицы через SQLAlchemy (как в текущем коде) и bootstrap-пользователя.

5. Запуск через **gunicorn** (слушает порт **8000** на всех интерфейсах — до него достучится только reverse proxy):

```bash
gunicorn -w 4 -b 127.0.0.1:8000 wsgi:app
```

Прокси (Caddy/nginx) принимает **443**, проксирует на **127.0.0.1:8000** и выставляет **`X-Forwarded-Proto: https`**.

**Включите** в окружении:

- `SAPPHIRE_BEHIND_PROXY=true`
- `SESSION_COOKIE_SECURE=true`

---

## Пример фрагмента Caddy (HTTPS + прокси)

Замените домен и путь к сокету/порту под себя:

```caddy
music.example.com {
    reverse_proxy 127.0.0.1:8000
}
```

Caddy сам получит сертификат Let's Encrypt.

## Пользователи «как в стриминге»

- **Только вы и знакомые:** оставьте регистрацию выключенной, раздайте логины вручную или один общий тестовый аккаунт (небезопасно, но для семьи иногда так).
- **Много людей сами себя регистрируют:** `SAPPHIRE_ALLOW_REGISTRATION=true` и продумайте модерацию/лимиты (в коде могут понадобиться доработки: капча, лимиты загрузки).

Помните: вы **ответственны за контент**, который хранится у вас на сервере (авторское право).

## Чем это всё ещё не Яндекс.Музыка

- Нет лицензированного каталога миллионов треков.
- Нет нативных приложений в сторах (есть **веб**; при желании можно обернуть в PWA или WebView).
- Масштабирование под миллионы слушателей — отдельная инженерная история (CDN, шардинг БД и т.д.).

Зато у вас **полный контроль**: свой домен, свои файлы, свои правила.

## Файлы в проекте, связанные с деплоем

| Файл | Назначение |
|------|------------|
| **`wsgi.py`** | Импорт `app` для gunicorn: `gunicorn wsgi:app` |
| **`requirements-prod.txt`** | `requirements.txt` + **gunicorn** |
| **`app.py`** | При `SAPPHIRE_BEHIND_PROXY` и `SESSION_COOKIE_SECURE` — корректная работа за HTTPS-прокси |

Локально по-прежнему: `python app.py` или `python desktop.py` — ничего не ломается, если прокси-переменные не заданы.

## Краткий чеклист «хочу сайт в интернете»

1. Арендовать VPS (Linux), привязать домен.  
2. Поставить Caddy или nginx + HTTPS.  
3. PostgreSQL + `DATABASE_URL`.  
4. Сильный `FLASK_SECRET_KEY`, `SESSION_COOKIE_SECURE=true`, `SAPPHIRE_BEHIND_PROXY=true`.  
5. `pip install -r requirements-prod.txt` и `gunicorn` за прокси.  
6. По возможности — S3/R2 для файлов.  
7. Решить вопрос регистрации и прав на музыку.

Дополнительно про API и загрузку с другого компьютера: [CLOUD-UPLOAD-RU.md](CLOUD-UPLOAD-RU.md).
